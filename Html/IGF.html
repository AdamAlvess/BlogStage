Réaliser une Analyse Fonctionnelle

La réalisation d'une analyse fonctionnelle permet de réfléchir et savoir quelle technologie est la plus adaptée pour réaliser ce logiciel et de voir la manière de tri la plus efficace selon la demande afin de ne pas perdre de temps lors de la réalisation du logiciel.

Cette section représente une part essentielle de mon analyse fonctionnelle, où j'examine en profondeur les technologies disponibles, leur application, leur pertinence, ainsi que la stratégie que j'adopterai pour réaliser mon logiciel.

   Pour coder mon application, j'ai dû choisir un langage que l'entreprise connaissait déjà. Parmi ceux qui m'ont été présentés, j'ai opté pour le C#. Ce langage est très complet et offre de nombreuses bibliothèques utiles pour mon projet, comme System.IO pour la gestion des entrées/sorties, notamment la manipulation des fichiers et des répertoires, et System.Windows.Forms pour créer des applications Windows Forms avec de nombreuses classes pour l'interface graphique, entre autres.

   En ce qui concerne la conception, j'ai opté pour une interface simple et intuitive. L'utilisateur doit simplement sélectionner le dossier contenant les fichiers à organiser et en utilisant une nomenclature de fichiers spécifique adaptée aux différents types de fichiers de l'entreprise, il sera possible de déplacer automatiquement chaque élément vers le bon emplacement. Il est également possible de choisir manuellement un autre chemin pour certains types de fichiers en décochant la case correspondante et en spécifiant le répertoire souhaité.

Codage de l'Application

J'ai développé quatre fichiers principaux pour la réalisation de ce projet. Le fichier IGF.cs contient le code de l'interface utilisateur pour l'application de gestion de fichiers en C#. Le fichier Trie.cs inclut la classe "Trie", qui regroupe toutes les fonctions nécessaires pour vérifier et trier les fichiers, notamment la fonction "StartSorting", qui est utilisée pour déclencher le tri. Le fichier IGF.Designer.cs gère le style de l'interface. Enfin, le fichier extract_account.cs contient la fonction "ExtractAccountNumbers", utilisée à plusieurs reprises dans la classe "Trie" pour traiter les factures. Et pour terminer j'ai réaliser un dossier assets ou il a était mis toute les images que j'ai utilisées.

Le code ci-dessous créer une interface utilisateur pour mon application de gestion de fichiers. Cette interface permet à l'utilisateur de sélectionner et trier différents types de documents à partir de dossiers spécifiques.

1. Structure générale :
Le code est organisé autour de la classe principale IGF, qui hérite de Form. Cette classe gère l'interface utilisateur et les interactions de l'utilisateur avec celle-ci.

2. Interface graphique :
L'interface est composée de plusieurs éléments comme des TextBox, Button, Label, et PictureBox. Par exemple, la TextBox sourceTextBox permet à l'utilisateur d'entrer le chemin source, tandis que les Button comme startButton ou browseButton permettent de démarrer le tri ou de parcourir les dossiers.

3. Sélection des types de documents :
Les différents types de documents (ODM, BL FACTURATION, FACTURE, etc.) sont gérés à travers des CheckBox et des TextBox associés. Chaque type de document a une case à cocher et une zone de texte où l'utilisateur peut indiquer le chemin spécifique.

4. Fonctionnalité de tri :
La méthode StartButton_Click est déclenchée lorsque l'utilisateur clique sur le bouton "Démarrer". Cette méthode appelle la fonction StartSorting de la classe Trie, en passant les chemins sélectionnés pour trier les documents selon les types spécifiés. La fonction GetDocumentTextBox est utilisée pour récupérer les chemins correspondants à chaque type de document.

5. Gestion des événements :
Des gestionnaires d'événements sont utilisés pour réagir aux actions de l'utilisateur. Par exemple, lorsque l'utilisateur coche ou décoche une case, la méthode CheckBox_CheckedChanged est appelée pour activer ou désactiver la TextBox associée.

6. Gestion des images :
Deux images, une en haut et une en bas, sont ajoutées à l'interface via les PictureBox headerPictureBox et footerPictureBox. Elles améliorent l'aspect visuel de l'interface.

Le code ci-dessous permet le trie des fichiers voulu. Le fichier Trie.cs est vraiment le back de mon application c'est la où toute ma logique ce trouve avec toute les étapes ce font après l'appuie du bouton "Démarrer".

La fonction StartSorting est utilisée pour trier et classer les fichiers d'un dossier source vers différents dossiers de destination en fonction de leur type. Elle commence par vérifier si le dossier source existe. Si ce n'est pas le cas, un message d'erreur est ajouté à la TextBox messageBox, et la fonction se termine.

Ensuite, la fonction parcourt tous les fichiers du dossier source. Pour chaque fichier, elle détermine son type en utilisant des fonctions spécifiques comme IsInvoiceFact pour les factures, IsInvoiceODM pour les ODM, etc. Une fois que le type du fichier est identifié, la fonction correspondante, par exemple ClassifyFact ou ClassifyODM, est appelée pour déplacer le fichier vers le dossier de destination approprié.

Si un fichier ne correspond à aucun des types prévus, un message est ajouté à la messageBox pour indiquer que le fichier n'a pas été classé. En cas d'erreur lors de l'exécution du tri, un message d'erreur est affiché, et la variable copySuccess est définie sur false. Si tous les fichiers sont triés avec succès et qu'aucun message d'erreur n'a été généré, un message de confirmation de la réussite de la copie est affiché dans la messageBox.

Pour le reste je vais expliquer pour les factures car c'est un type de fichier spécifique mais pour le reste j'expliquerai seulement un pour avoir la logique de tous.

La fonction ClassifyFact est utilisée pour classifier et déplacer les fichiers de type "facture" vers un répertoire de destination approprié. Elle prend en paramètre un objet FileInfo représentant le fichier à classer, une TextBox pour afficher les messages d'erreur ou de réussite, une CheckBox qui détermine si le fichier doit être traité, et enfin le chemin du répertoire de destination.

Si la CheckBox correspondant aux factures est cochée, la fonction commence par extraire le numéro de compte à partir du fichier en utilisant la méthode ExtractAccountNumbers. Ce numéro de compte est ensuite utilisé pour déterminer le chemin de destination approprié via la méthode GetDefaultFactureDestination. Si un chemin de destination valide est trouvé, le fichier est copié vers ce répertoire. En cas d'échec, un message d'erreur est affiché dans la TextBox.

Si la case n'est pas cochée, la fonction vérifie si un chemin de destination alternatif est spécifié. Si ce n'est pas le cas, un message d'erreur est affiché et le processus est interrompu. Si le fichier existe, il est copié vers ce chemin alternatif. Si le fichier n'existe pas, un message d'erreur est ajouté à la TextBox pour informer l'utilisateur.

La fonction ClassifyBLFO est conçue pour classifier et déplacer les fichiers de type "Bon de Livraison Fournisseur" (BLFO) vers un répertoire de destination spécifique. Elle prend en paramètres un objet FileInfo représentant le fichier à traiter, une TextBox pour afficher les messages d'erreur ou de confirmation, une CheckBox qui indique si le fichier doit être traité, et enfin, le chemin de destination pour les fichiers BLFO.

Si la CheckBox correspondant aux BLFO est cochée, la fonction commence par extraire le numéro de compte du fichier à l'aide de la méthode ExtractAccountNumbers. Ensuite, ce numéro est utilisé pour déterminer le chemin de destination approprié via la méthode GetDefaultBLFODestination. Si un chemin valide est trouvé, le fichier est copié vers ce répertoire. En cas d'échec, un message d'erreur est affiché dans la TextBox pour informer l'utilisateur.

Si la CheckBox n'est pas cochée, la fonction vérifie si un chemin de destination alternatif pour les BLFO a été spécifié. Si aucun chemin n'est fourni, le processus est interrompu et un message d'erreur est affiché. Si le fichier existe, il est copié vers ce chemin alternatif. Si le fichier n'existe pas, un message d'erreur est ajouté à la TextBox pour alerter l'utilisateur.

La fonction IsInvoiceFact est utilisée pour déterminer si un fichier donné est une facture ou un avoir. Elle prend en entrée un objet FileInfo représentant le fichier et une CheckBox associée. La fonction convertit le nom du fichier en minuscules pour faciliter la recherche et vérifie ensuite si le nom du fichier contient les chaînes "fact" ou "avoir". Si l'une de ces chaînes est présente, la fonction retourne true, indiquant que le fichier est bien une facture ou un avoir.

De manière similaire, la fonction IsInvoiceDevis vérifie si un fichier est un devis. Elle suit la même logique que IsInvoiceFact, mais elle recherche cette fois-ci la chaîne "devis" dans le nom du fichier. Si le nom du fichier contient "devis", la fonction retourne true, confirmant ainsi qu'il s'agit bien d'un devis.

Pour les autres types de fichiers, la même logique est appliquée. Chaque fonction vérifie si le nom du fichier contient une chaîne spécifique correspondant au type de document recherché. Par exemple, pour identifier un bon de livraison fournisseur (BLFO), la fonction vérifiera la présence de termes spécifiques dans le nom du fichier, en suivant le même principe que IsInvoiceDevis.

La fonction GetDefaultFactureDestination est conçue pour déterminer le chemin de destination par défaut où une facture doit être copiée. Cette fonction prend en entrée un numéro de compte (accountNumber) et une TextBox (messageBox) pour afficher des messages.

Elle commence par obtenir le chemin du dossier utilisateur local à l'aide de la méthode Environment.GetFolderPath et concatène ce chemin avec le chemin spécifique EKIUM\7C27 - D-EXECUTE pour accéder au dossier D-Execute.

Ensuite, elle parcourt tous les sous-dossiers du dossier D-Execute à l'aide de la méthode Directory.GetDirectories. Pour chaque dossier, elle extrait le nom du dossier et le divise en utilisant le caractère tiret ("-"). Le premier élément du résultat correspond au numéro de compte du dossier. Si ce numéro de compte correspond à celui fourni en entrée, la fonction construit le chemin de destination en ajoutant le chemin relatif A02-Planification - Suivi d'affaire\FIN-SUIVI FINANCIER\FC-Facture au chemin du dossier correspondant. Ce chemin de destination est ensuite renvoyé par la fonction.

Si aucun dossier correspondant n'est trouvé, la fonction affiche un message d'erreur dans la TextBox et retourne null, indiquant qu'aucune destination appropriée n'a été trouvée.

La fonction ExtractAccountNumbers sert à extraire le numéro de compte à partir du nom d'un fichier de facture. Elle prend en entrée le chemin complet du fichier (filePath) et une TextBox (messageBox) pour les messages.

La fonction commence par vérifier si le fichier existe à l'aide de File.Exists. Si le fichier n'existe pas, elle retourne null. Si le fichier existe, elle récupère le nom du fichier sans l'extension grâce à la méthode Path.GetFileNameWithoutExtension.

Ensuite, elle cherche la position du premier tiret ("-") dans le nom du fichier. Si un tiret est trouvé, la fonction extrait la partie du nom de fichier située avant ce tiret, qui est supposée être le numéro de compte. Ce numéro est alors retourné.

Si aucun tiret n'est trouvé dans le nom du fichier, la fonction affiche un message d'erreur dans la TextBox et retourne null, indiquant qu'il n'a pas été possible d'extraire le numéro de compte.

La fonction GetDefaultBLFODestination est conçue pour localiser le chemin de destination par défaut où un bon de livraison fournisseur (BLFO) doit être copié. Elle prend en entrée un numéro de compte (accountNumber) et une TextBox (messageBox) pour afficher des messages d'erreur ou d'information.

La fonction commence par obtenir le chemin du dossier utilisateur local en utilisant la méthode Environment.GetFolderPath et combine ce chemin avec EKIUM\7C27 - D-EXECUTE pour accéder au dossier D-Execute, qui est supposé contenir les projets associés à différents comptes.

Ensuite, la fonction parcourt tous les sous-dossiers à l'intérieur du dossier D-Execute à l'aide de la méthode Directory.GetDirectories. Pour chaque sous-dossier, elle extrait le nom du dossier, qui est supposé être composé d'un numéro de compte suivi d'un tiret ("-") et du nom de l'affaire (projet). Le nom du dossier est alors divisé en deux parties en utilisant le tiret comme séparateur.

La première partie de ce nom de dossier est supposée être le numéro de compte. Si ce numéro de compte correspond à celui fourni en entrée, la fonction construit le chemin de destination pour le bon de livraison en ajoutant le chemin relatif A04-Achats\PRJ-PROJET\BL-Bon de livraison au chemin du dossier correspondant. Ce chemin de destination est alors retourné par la fonction.

Si aucun dossier correspondant n'est trouvé, la fonction affiche un message d'erreur dans la TextBox pour indiquer qu'aucun dossier ne correspond au numéro de compte fourni, et retourne null.
